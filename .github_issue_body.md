## Bug: False "Field not found in current view" for views that extend another view

### Description

When a view uses `extends: [base_view]` (or `include` + a refining view), the validator reports **"Field 'X' not found in current view 'Y'"** for dimensions/measures that exist on the extended (base) view. Unqualified references like `${day_date}` or `drill_fields: [day_date, ...]` are valid in LookML in this case but are incorrectly flagged.

### Example

- **Base view** `_global_sales_aggregated_raw`: defines `dimension_group: day` (so `day_date` exists) and dimensions like `ecosystem`, `country_id`, etc.
- **Refining view** `global_sales_aggregated`: has `extends: [_global_sales_aggregated_raw]` and uses:
  - `drill_fields: [day_date, ecosystem, customer, ...]`
  - In SQL: `${day_date}`, `${ecosystem}`, etc.

The language server reports errors such as:
- `Field "day_date" not found in current view "global_sales_aggregated"`

These are false positives: in LookML, a view that extends another view inherits its fields, so unqualified references to those fields in the refining view are correct.

### Expected behavior

- References to dimensions/measures (and dimension_group timeframes like `day_date`) that exist on the current view **or on any view it extends** should be accepted.
- No error when using unqualified names like `day_date` in a view that extends a view where `day_date` is defined (e.g. via `dimension_group: day` with timeframe `date`).

### Root cause (summary)

The diagnostics only check the current view’s own `dimension` / `measure` / `dimension_group` and do not resolve `extends`. The workspace model merges fields only for refinement views whose names start with `+`, not for views using `extends: [view_name]`.

### Possible fix

Introduce a helper that resolves “effective” fields for a view by following `extends` (and handling dimension_group timeframes). Use it wherever the validator checks that a field exists in a view (drill_fields, SQL refs, set fields, join sql_on).
