## Bug: LSP does not detect circular references in LookML SQL

### Description

The LookML language server validates `${view.field}` and `${field}` references in dimension/measure/dimension_group SQL and reports "field not found", but it does **not** detect circular references.

Example: a dimension that references itself is valid LookML syntax and passes current validation, but Looker will fail at runtime with a circular reference error.

```lookml
view: global_sales {
  dimension: publisher {
    type: string
    sql: ${publisher} ;;
    label: "Publisher"
  }
}
```

**Expected:** The LSP should report a diagnostic such as:  
`Circular reference detected [view_name, view_name.field_name].`

**Actual:** No diagnostic; the circular reference is only reported when validating in Looker (e.g. "Circular reference detected [monthly_forecast, global_sales.publisher]").

### Impact

Users may only discover circular references after pushing to Looker instead of getting immediate feedback in the editor.

### Proposed fix

- Add a dedicated diagnostic (e.g. `SQL_REF_CIRCULAR`) for circular references.
- For each view, build a dependency graph of fields based on SQL references (same-view `${field}` and `${view_name.field}`).
- Use a cycle-detection algorithm (e.g. DFS with three-color marking) to find cycles and report them on the offending SQL property.
