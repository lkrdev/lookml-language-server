## Summary

Fixes false **"Field 'X' not found in current view 'Y'"** diagnostics when a view uses `extends: [base_view]`. Fields inherited from the extended view (and dimension_group timeframes like `day_date`) are valid in the refining view but were not recognized.

## Changes

- **`fieldExistsInView(viewName, fieldName, visited?)`**  
  New private helper that:
  - Considers dimensions, measures, and dimension_group from the current view.
  - Handles **dimension_group timeframes** (e.g. `day_date` from `dimension_group: day` with timeframe `date`).
  - Resolves **`extends`** recursively (with cycle detection via `visited` set).
  - Returns true if the field exists on the view or any view it extends.

- **All “field not found in view” checks** now use this helper:
  - `validateDrillFields` (drill_fields array)
  - `validateSqlReferences` (both `${view.field}` and `${field}`)
  - Set field validation (fields in a set)
  - Join `sql_on` validation (replaced the previous inline `viewHasField` logic, which also had a bug: it used `fieldName` instead of the `field` parameter)

## Testing

- Manual check with a view that extends another view and uses unqualified refs (e.g. `drill_fields: [day_date]`, `${day_date}` in SQL): the previous “field not found” errors should disappear.
- No new tests added; existing tests should remain pass (project has no dedicated diagnostics tests for this path).
